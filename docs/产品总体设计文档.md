## 一、文档概述

### 1.1 项目名称

**景区导览服务 AI 数字人** — 基于多模态大模型的智能景区导览系统

### 1.2 项目背景

随着文旅产业数字化转型加速，游客对个性化、沉浸式游览体验的需求日益增长。传统景区导览存在导游人力有限、讲解内容单一、多语言支持不足、游客反馈难以量化等痛点。本项目以无锡灵山胜境为示范景区，构建基于 AI 数字人技术的智能导览系统，为游客提供 7×24 小时多模态交互服务，为景区管理方提供数据驱动的运营决策支持。

### 1.3 文档用途

本文档面向评审专家，全面描述系统的架构设计、技术选型、模块划分及核心实现方案。

## 四、核心模块设计

### 4.1 Brain 大脑层

#### 4.1.1 意图分流引擎 (`brain/pipeline.py`)

```
用户输入 → 关键词匹配 → 路由分发:
  ├── 路线类 (路线/怎么逛/一日游) → _handle_route()
  ├── 知识类 (历史/门票/开放/建筑) → _handle_rag()
  └── 闲聊类 (其他) → _handle_chat()
```

**关键词库**：27 个 RAG 触发词 + 17 个路线触发词，覆盖主流景区咨询场景。

#### 4.1.2 RAG 知识检索 (`brain/rag/`)

**三阶段检索流程**：

```
阶段1: 向量检索
  用户问题 → BGE-M3 嵌入 → ChromaDB top_k=12 候选
    ↓
阶段2: LLM 重排
  12 候选 → Qwen Omni 精选 3-4 条最相关
    ↓
阶段3: 知识增强生成
  精选上下文 + System Prompt → LLM 生成回答
```

**知识库数据**（赛题示范资料包）：
- `灵山胜境：历史、文化、景点特色与个性化游览指南.txt` (17KB)
- `灵山胜境 景点结构化数据集.txt` (49KB)
- `灵山游客消费统计.txt` (1.6KB)
- 总计分块：142 个语义块

**评估框架** (`brain/rag/evaluate.py`)：
- 51 题 QA 测试集，关键词 OR 匹配评分
- 目标准确率 ≥ 90%

#### 4.1.3 ASR 语音识别 (`brain/server.py`)

- **模型**：OpenAI Whisper base（74MB），中文优化
- **优化**：模型常驻内存（启动时预加载），消除每次请求 ~4 秒冷启动
- **领域适配**：注入景区专有名词做语言模型偏置（`initial_prompt`）
- **纠错**：50+ 组同音词纠错映射（如"林山大佛"→"灵山大佛"）
- **适配**：16kHz 重采样 + 峰值归一化

#### 4.1.4 多模态 LLM 客户端 (`brain/llm_client.py`)

- 支持文本输入和图像输入（base64 编码）
- DashScope MultiModalConversation API
- 流式输出：`chat_stream()` 生成器，SSE 推送

#### 4.2.2 分析引擎 (`admin/backend/app/analyzer.py`)

- **关键词提取**：jieba 分词 + 自定义停用词 + 景区专有名词白名单（30+ 词）
- **情感分析**：基于关键词匹配（正面 30+ 词，负面 20+ 词）
- **时段分布**：按小时统计咨询量，识别高峰低谷
- **主题分类**：6 大主题（门票价格、交通出行、景点特色、餐饮服务、游玩攻略、表演活动）
- **智能建议**：规则引擎，根据关键词 + 情感比例生成运营建议

### 4.3 LiveTalking 数字人渲染层

#### 4.3.1 技术方案

- **模型**：wav2lip256（205MB），基于 GAN 的口型同步模型

## 五、关键技术决策

### 5.1 为什么选择 Qwen Omni

| 对比维度 | Qwen Omni | GPT-4o | DeepSeek-V3 |
|---------|-----------|--------|-------------|
| 多模态 | ✅ 文本+图像 | ✅ 文本+图像 | ❌ 仅文本 |
| 中文能力 | 原生中文训练 | 多语言 | 原生中文训练 |
| 生态 | 阿里云 DashScope | OpenAI API | 独立 API |
| 成本 | 免费额度充足 | 需付费 | 低价 |
| 赛题匹配 | 国产模型 | 国外 | 国产模型 |

选择 Qwen Omni 的核心原因：(1) 支持多模态，满足赛题硬性要求；(2) 原生中文能力；(3) 国产技术栈。

### 5.2 为什么用 wav2lip 而非 MuseTalk

| 对比维度 | wav2lip | MuseTalk |
|---------|---------|----------|
| 模型大小 | 205MB | ~2GB |
| 推理速度 (RTX 3060) | 60 FPS | 42 FPS |
| 显存占用 | ~2GB | ~6GB |
| 部署难度 | 低 | 中 |
| 形象素材 | 550 帧即可 | 需完整视频 |

选择 wav2lip 的原因：(1) 轻量，RTX 4060 可流畅运行；(2) 推理速度快；(3) 形象制作成本低。

### 5.3 为什么管理前后端合并到单进程

原设计是 Brain(8011) + Admin(8012) 两个独立进程。考虑到：(1) 简化部署；(2) 管理后台 API 与 Brain 共享数据库和模型实例；(3) 减少端口占用。将 Admin API 通过 FastAPI router 挂载到 Brain 进程，前端静态文件通过 StaticFiles 挂载。

## 六、数据流图

### 6.1 文本问答流程

```
浏览器 → POST /ask {text, sessionid}
  → pipeline.handle_user_input()
    → intent 路由 (route/rag/chat)
      → rag: ChromaDB 检索 → LLM 知识增强生成
      → route: 路线推荐引擎 → LLM 润色
      → chat: LLM 直接对话
    → 写入 ChatLog (SQLite)
    → 推送 LiveTalking (/human)
  → 返回 {reply, intent, latency_sec}
```

### 6.2 语音问答流程

```
浏览器 → MediaRecorder 录音 → AudioContext 转 WAV
  → POST /asr {file}
    → Whisper base 转写 → 同音词纠错 → 简繁转换
  → 浏览器拿到 text
  → POST /ask {text}
  → (同上文本问答流程)
```

### 6.3 拍照识景流程

```
浏览器 → input[type=file] 拍照
  → POST /upload-image {file} → 服务器保存 → 返回 image_path
  → POST /ask {text, image_path}
    → LLM 接收 text + base64 图片 → 多模态理解 → 生成回答
```
